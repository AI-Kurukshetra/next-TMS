# NextGen TMS Project Instructions

## Project Context
- Product: `NextGen Transportation Management System`
- Stack: `Next.js 15 App Router`, `Supabase`, `Tailwind CSS`, `TypeScript`
- Scope: hackathon MVP, but code should still be structured for extension rather than throwaway demos

## Core Principles
- Prefer working code over placeholder scaffolding.
- Keep the repo shippable: every new route should have a real loading, empty, and error path.
- Use the existing repository pattern instead of querying Supabase directly from random components.
- Preserve the current architecture unless there is a clear reason to move it.
- When changing schema, update both `supabase/migrations/` and `database/schema.sql`.

## Architecture Rules
- `src/app/`: routing, layouts, pages, route handlers only
- `src/components/`: presentational UI and client interaction components
- `src/lib/repositories/`: data access and domain queries
- `src/lib/validations/`: payload parsing and basic validation
- `src/types/`: shared domain types
- Avoid putting business logic in page components when it can live in repositories or helpers.
- Prefer server components by default. Use client components only for interactivity, auth state listeners, or browser-only APIs.

## Supabase Rules
- Do not create tables manually in the dashboard one by one.
- All schema changes must be expressed in SQL migrations.
- Reuse shared Supabase helpers from `src/lib/supabase/`.
- Handle missing-table and query errors explicitly; do not let raw PostgREST failures crash the entire page.
- If auth behavior changes, consider impact on both browser session state and API/server access.

## Authentication Rules
- Auth must be functional, not mocked.
- Protect dashboard routes from unauthenticated access.
- Expose login, signup, and logout in the UI.
- Do not assume auth is enforced unless both page access and API access are covered.
- Keep role support aligned with database enums and user profile records.

## UI and Design Rules
- Avoid generic admin-template output.
- Keep the visual language consistent: clean logistics dashboard, high readability, restrained color use, strong spacing.
- Every page should have:
  - a clear page header
  - at least one meaningful primary action or data surface
  - usable empty states
- Forms should include labels, disabled states, submission feedback, and actionable error text.
- Tables should prioritize operational scanning: IDs, statuses, route/location, assignee, timestamps.
- Do not ship dead buttons, dead navigation, or copy that describes missing functionality as if it exists.

## Coding Best Practices
- Prefer small, composable functions.
- Use explicit types for domain objects and route payloads.
- Reuse utilities before adding duplicates.
- Keep naming consistent with the transportation domain: `shipment`, `carrier`, `driver`, `vehicle`, `route`, `tracking event`.
- Avoid hidden magic values; centralize status lists and option sets when reused.

## Database and Data Flow
- Primary write path: form or API -> validation -> repository -> Supabase
- Primary read path: page/route -> repository -> typed result -> UI
- Seed data is acceptable for MVP demos, but production assumptions must not depend on seeded rows.

## Definition of Done
- Feature is wired end to end.
- UI path is reachable from navigation or direct route.
- Data persists or reads correctly from Supabase when applicable.
- TypeScript passes.
- If schema changed, migration SQL and bootstrap SQL are updated.

## Design Quality Bar
- Make interfaces feel intentional, not autogenerated.
- Prefer simple layouts with strong hierarchy over crowded dashboards.
- Use visual emphasis for shipment status, exceptions, and assignment state.
- Optimize for dispatch workflow first: speed to scan, assign, and update.
